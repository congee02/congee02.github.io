JVM 执行一个方法的流程 （add 方法）
    1. 创建栈帧
    2. 保存 main 方法的程序计数器
    3. 修改线程的局部表开始指针为 add 方法的
    4. 修改线程的操作数栈开始指针为 add 方法的
add 执行完了，JVM底层是怎么做的
    1. 修改线程的操作数栈开始指针为 main 方法的
    2. 修改线程的局部表开始指针为 main 方法的
    3. 恢复 main 方法的程序计数器
    4. 判断有无返回值，若有，则压入 main 方法的操作数栈
    5. 释放栈帧

Preview - JVM GC

  1. 判定垃圾算法 - Root Searching，根可达

    根可达性算法是JVM中一种判断对象是否回收的算法，基本思路就是通过一系列名为”GC Roots”的对象作为起始点，从这些节点开始向下搜索，搜索所走过的路径称为引用链(Reference Chain)，当一个对象到GC Roots没有任何引用链相连时，则证明此对象是不可用的。

     

  2. GC 算法

    1. Mark-Sweep
	将标记的内存释放 -> 内存碎片化的
2. Copying
	将未标记且使用的内存复制到另外一个大空的内存块 -> 内存浪费
3. Mark-Compact
	清除后整理碎片 -> 效率较低

  3. 堆内存逻辑分区

    发生在新生代的 GC 称为 YGC，发生在老年代的称为 OGC，上述两者都发生称为 FGC

     

  4. YGC

    将新诞生的对象置于伊甸园区。第一次 YGC，From Survivor 和 To Survivor 都为空，在 Eden 区 GC 时，将活着的对象复制到 From Survivor；第二次 YGC，Eden 和 From Survivor 都可能不为空，To Survivor 为空，将两区存活的对象复制到 To Survivor；YGC，Eden 和 To Survivor 都可能不为空，From Survivor 为空，将两区存活的对象复制到 From Survivor。
YGC 非常频繁，因此要求 YGC 效率高，整理碎片的效率比 copy 低。

  5. 如果 Survivor 不够大

    统计得出： YGC 一般能回收 90 % 的垃圾。
如果Survivor区放不下存活对象，存活对象并不是全都进入老年代，而是部分对象进入老年代，部分对象继续被分配到Survivor区。

  8. 三色标记算法 

    三种颜色对应了对象的三种状态
	1. 黑色：对象本身已经被标记，且下属的属性也被全部标记过了
	2. 灰色：对象本身已经被标记，但下属的属性未被全部标记过，GC 需要从此对象的属性中去寻找垃圾
	3. 白色：该对象没有被标记过。


    扫描前存在 B->D 的引用。在下一次扫描之前，B -> D 的引用消失，D 变为垃圾，此时 GC 不认为 D 为 垃圾，此时D被称为 “浮动垃圾”，等待下一次 GC 清除。
若老年代堆内存占 90 %，等到 GC 时，新生代产生的垃圾会将剩余的 10 % 填满，导致堆内存占用 100 %，触发 STW，所有的业务进程停止，


    GC 发现 B->D 消失，未发现 A->D 增加。
此时，GC 认为 D 为 垃圾，然而 A 还在引用 D。若不加以任何处理，GC 回收 D，则 A.d = null, 导致 NPE。
解决：屏障，执行 A.d = d 时，AOP，若 A 为 黑色，D 为白色，标 A 为灰。称为 增量更新，Incremental Updates
Bug: 并发标记，产生漏标，需要 remark 阶段，从头扫描一遍

  9. 浮动垃圾

    remark过程标记活着的对象，从GCRoot的可达性判断对象活着，但无法标记“死亡”的对象。
如果在初始标记阶段被标记为活着，并发运行过程中“死亡”，remark过程无法纠正，因此变为浮动垃圾，需等待下次gc的到来。

  10. CMS, Concurrent Mark Sweep，并发垃圾回收

    内存越大，越不适合  CMS
缺点：
1. Floating Garbage 浮动垃圾 STW
2. Remark STW

     

  11. SATB，Snapshot At the Beginning

    当 "灰"->"白" 消失，或者 当出现 "黑"->"白" 时，将引用记录到一个堆栈中。

下一次 GC 扫描时，检查堆栈，若堆栈非空，说明有 "灰"->"白" 消失的情况。此时检查 "白"，若查询堆栈中没有任何其他可达对象引用 "白"，则判定该 "白"对象 为垃圾。

怎么知道有任何其他可达对象指向 "白"对象 ？

     

  12. G1 GC 内存布局

    三色标记算法 + SATB -> GC 并发

G1 将内存分区，分为一个个 Region，当某个 Region 快满了，则清空该 Region。新生代，老年代可以按需要动态地被分配空的 Region。

     

  13. G1 调优

    1. 追求吞吐量
	100 cpu, 99 app 1 GC，吞吐量 = 99 %
2. 追求响应时间
	XX:MaxGCPauseMillis 200 控制 STW

  14. G1 是灵活的

    1. 分 Region 回收
2. 优先回收 花费时间少、垃圾比例高的 Region

  15. G1 动态调整各个分区，谨慎手动各个分区的比例
